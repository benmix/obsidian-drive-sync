# 编码规范

## 1. 目的

本文档定义 Obsidian 云盘同步 的仓库级编码规范。

它不是通用 TypeScript 教程，目标是让代码变更与当前仓库的同步架构、运行时边界和插件行为保持一致。

## 2. 核心原则

- 可读性优先于技巧。
- 优先扩展现有流程，而不是先引入新抽象。
- provider 专用逻辑不要进入 provider 无关层。
- 对同步、状态和错误处理这类运行时行为保持显式表达。
- 以真实插件的 reload、retry、recovery 场景下的可维护性为目标。

## 3. 分层边界

源码位于 `src/`，应遵循当前分层：

- `src/main.ts`：最小化插件入口
- `src/runtime/`：生命周期编排、协调、会话控制、面向应用的运行时行为
- `src/sync/`：provider 无关的同步规划、执行、状态流转和同步支撑逻辑
- `src/provider/`：provider 专用集成和 SDK 适配
- `src/ui/`：设置页、弹窗和用户界面渲染
- `src/contracts/`：共享契约和跨层类型
- `src/data/`：持久化和 IndexedDB 访问
- `src/errors/`：结构化错误系统和归一化辅助函数
- `src/i18n/`：翻译和本地化辅助

边界规则：

- `runtime/` 不能依赖 `ui/` 的实现细节，也不能直接依赖具体 provider 模块。
- `provider/` 不能依赖 `runtime/` 或 `sync/` 内部实现。
- `sync/` 必须保持 provider 无关。
- `ui/` 可以消费 runtime-facing contract，但不应拥有 sync engine 行为。
- 共享类型应进入 `contracts/`，不要通过隐藏的跨层导入来复用。

如果某个改动看起来必须打破这些边界，先重新审视设计。

## 4. 文件与模块设计

- 保持 `src/main.ts` 精简，只承担插件启动、注册和卸载。
- 优先使用小而聚焦的模块，而不是大而全的文件。
- 如果一个文件同时混合编排、持久化、UI 格式化和 provider 行为，就应拆分。
- 新模块应放到真正拥有该行为的层附近，不要创建泛化的 `utils` 桶。
- 当现有 use-case 或 service 模块可扩展时，不要再引入平行抽象。

建议拆分的信号：

- 文件已经无法在一次 review 中清晰看完
- 多个职责因为不同原因同时变化
- 逻辑拆出后测试会明显更清晰
- import 关系表明该文件跨了多个层级

## 5. TypeScript 规则

- 保持 TypeScript 严格模式。
- 优先使用显式的领域类型，不要依赖松散 record 或临时对象字面量。
- 对未知错误或外部输入边界使用 `unknown`，然后做归一化。
- 对跨模块数据优先使用收窄 union、具名 interface 或 type alias。
- 除非存在明确的外部约束并且有说明，否则避免 `any`。
- 不需要变更的数据优先使用 `readonly`。

## 6. 导入与依赖

- 优先从定义模块直接导入。
- import 必须遵守分层模型，不要随意跨架构引用。
- 避免循环依赖。如果两个模块互相需要，把共享契约下沉到 `contracts/` 或重构归属。
- 使用仓库脚本和 lint 规则发现导入顺序与分层问题，而不是在 review 里口头约束。

本仓库相关验证工具：

- `pnpm run lint`
- `pnpm run lint:fix`
- `pnpm run lint:layers`
- `pnpm run fix:imports`

## 7. 异步与副作用

- 优先使用 `async` / `await`，避免过长的 promise chain。
- 副作用应尽量靠近编排边界，便于推理。
- 文件系统、网络或持久化写入不要隐藏在命名含糊的通用 helper 里。
- 插件启动阶段避免不必要的重工作，能延后就延后。
- 对文件系统触发的工作尽量做 debounce 或批处理。

## 8. 错误处理

- 新代码应优先使用 `src/errors/` 中的结构化错误，而不是继续依赖字符串匹配。
- 外部 SDK 或 `unknown` 错误在参与运行时决策前应先归一化。
- 用户可见文案和诊断细节必须分离。
- 不要在普通 UI 中暴露凭据、原始 provider 标识或不安全的内部细节。
- 如果失败会改变 runtime state，要在代码里显式表达该状态流转。

更完整的设计参见 [`docs/ERROR_SYSTEM_DESIGN.md`](./ERROR_SYSTEM_DESIGN.md)。

## 9. UI 与用户文案

- UI 文案保持简短、直接、一致。
- 标题、按钮和 section label 优先使用 sentence case。
- 普通 UI 中展示对用户安全的错误信息；原始技术细节保留给诊断。
- 修改状态逻辑时，要同时更新状态语义和展示层。
- 除非任务明确要求重设计，否则遵循现有视觉语言。

## 10. 持久化与状态

- 把插件持久化状态和同步状态视为稳定契约。
- 除非有明确迁移方案，否则优先采用增量式 schema 演进，而不是破坏式重写。
- 持久化字段命名应稳定，并表达明确行为语义。
- 区分临时 runtime state 和持久化 recovery state。
- 新增 retry、blocked、auth 相关状态时，要明确恢复语义。

## 11. 测试要求

- 非平凡逻辑改动应新增或更新定向测试。
- 对纯同步逻辑、planner、状态流转和错误归一化优先写单元测试。
- 对代码改动，`pnpm run build` 是最基本的广覆盖验证。
- 当改动影响结构、import、风格或跨模块边界时，运行 `pnpm run lint`。
- 如果跳过了相关检查，要在变更说明里明确指出。

## 12. 文档要求

- `docs/` 下的用户文档必须与 `docs/zh-CN/` 下的对应文件保持双语同步。
- 当行为、流程或约束变化时，同步更新文档。
- 优先使用与当前仓库一致的说明，不要回退到泛化 sample-plugin 文案。
- 示例代码和说明应与当前仓库结构保持一致。

## 13. 实务 Do / Don't

Do：

- 复用现有 runtime 和 sync 流程。
- 把 provider 专用行为隔离在 `provider/`。
- 保持插件生命周期代码精简。
- 让状态流转和重试语义一眼可见。
- 写出能承受 reload、retry 和部分失败场景的代码。

Don't：

- 把 provider 专用逻辑放进 `sync/`。
- 没有明确要求时引入兼容胶水代码。
- 把副作用偷偷塞进格式化或通用 helper 函数。
- 在新代码里继续依赖 message 文本做核心运行时决策。
- 引入仓库其余部分并不需要的宽泛抽象。
