# 编码规范

## 1. 目的

这份文档定义 Obsidian Drive Sync 的仓库专用工程规则。它不是通用 TypeScript 风格指南，重点是让改动始终符合当前仓库的同步模型、运行时边界和恢复行为。

## 2. 核心原则

- 清晰优先于炫技。
- 优先扩展现有流程，而不是增加新抽象。
- provider 专属行为不要混入 provider 无关层。
- 状态迁移、重试和失败处理必须显式表达。
- 代码要在插件重载、重试和部分失败场景下仍然可维护。

## 3. 分层边界

源码位于 `src/`，按下面的职责划分：

- `src/main.ts`：插件门面、生命周期接线与委派
- `src/runtime/`：编排、会话、调度与面向应用的运行时行为
- `src/sync/`：provider 无关的同步规划、执行和状态行为
- `src/provider/`：provider 专属集成与 SDK 适配
- `src/ui/`：设置页、弹窗与用户可见视图
- `src/contracts/`：共享契约和跨层类型
- `src/data/`：持久化与 IndexedDB 访问
- `src/errors/`：结构化错误和规范化辅助函数
- `src/i18n/`：翻译与本地化辅助工具

边界规则：

- `runtime/` 不能依赖具体 provider 实现或 UI 内部实现。
- `provider/` 不能依赖 `runtime/` 或 `sync/` 内部细节。
- `sync/` 必须保持 provider 无关。
- `ui/` 可以依赖插件面向外部的契约，但不能拥有同步引擎行为。
- 共享契约应放入 `contracts/`，不要靠隐式跨层导入维持。

如果一个改动看起来必须打破这些边界，先重新审视职责归属。

## 4. 文件与模块设计

- 保持 `src/main.ts` 小而明确。
- 优先使用职责单一的模块，不要把过多行为塞进一个文件。
- 如果一个文件同时混合编排、持久化、UI 格式化和 provider 行为，就应该拆分。
- 新模块应放在真正拥有该行为的层里。
- 不要把领域逻辑塞进泛化的 `utils` 目录。

通常满足以下条件之一就该拆文件：

- 一次 review 很难把文件看明白
- 不相关的改动总落在同一个文件里
- 如果拆开，测试会更清晰
- 导入列表已经跨越太多架构层

## 5. TypeScript 规则

- 保持严格类型设置。
- 优先显式领域类型，而不是松散对象和临时字面量。
- 在不安全边界使用 `unknown`，随后立刻规范化。
- 跨模块数据优先使用窄联合和具名接口。
- 除非外部依赖逼迫，否则避免 `any`。
- 不需要修改的数据尽量使用 `readonly`。

## 6. 导入与依赖

- 从真正拥有该逻辑的模块导入，而不是依赖模糊的转发导出。
- 导入关系要服从分层模型。
- 避免循环依赖。如果两个模块共享概念，应把契约下沉。
- 让仓库脚本和检查规则负责约束结构。

仓库检查命令：

- `pnpm run lint`
- `pnpm run lint:fix`
- `pnpm run lint:layers`
- `pnpm run fix:imports`

## 7. 异步与副作用

- 优先使用 `async` / `await`，不要堆长链式 Promise。
- 副作用尽量靠近编排边界。
- 不要把文件系统、网络或持久化写操作藏在命名模糊的辅助函数里。
- 如果可以安全延后，就不要在插件启动阶段做重活。
- 文件系统触发的工作尽量做防抖或批处理。

## 8. 错误处理

- 新代码应优先使用 `src/errors/` 中的结构化错误。
- provider 或 SDK 抛出的错误要先规范化，再参与运行时决策。
- 用户可见文案和诊断细节要分开。
- 常规 UI 不得泄露认证信息或不安全的 provider 细节。
- 失败导致状态改变时，要在代码里显式表达该状态迁移。

系统级设计见 [`docs/ERROR_SYSTEM_DESIGN.md`](./ERROR_SYSTEM_DESIGN.md)。

## 9. UI 与用户文案

- 文案要短、直接。
- 标题、按钮和标签使用句式大小写，不要过度装饰。
- 普通 UI 中优先显示安全且可翻译的错误消息。
- 状态语义改变时，要同时更新状态模型和展示逻辑。
- 除非任务明确要求重做设计，否则保持现有 UI 语言一致。

## 10. 持久化与状态

- 把持久化字段视为稳定契约。
- 优先做增量 schema 变更，除非有明确迁移方案。
- 字段命名要稳定且有明确行为含义。
- 区分临时运行时状态与必须跨重载保留的恢复状态。
- 对重试、阻塞和认证相关状态，要明确它们的恢复语义。

## 11. 测试要求

- 非平凡逻辑改动必须补充或更新针对性测试。
- 同步规划、状态迁移和错误规范化优先做单元测试。
- `pnpm run build` 是代码改动最基本的广义验证。
- 结构、导入和分层发生变化时，应运行 `pnpm run lint`。
- 如果跳过了相关检查，要在变更说明里写明。

## 12. 文档要求

- `docs/` 下的用户可见文档必须保持双语同步。
- 行为、流程或架构约束变化时，同步更新文档。
- 优先写仓库专用说明，不要复用 sample plugin 套话。
- 例子要与当前代码结构一致。

## 13. 实务 Do / Don't

应该做：

- 复用现有 runtime 和 sync 流程
- 把 provider 专属行为隔离在 `provider/`
- 保持生命周期接线简洁
- 让重试和恢复逻辑一眼可见
- 写出能扛住重载和部分失败的代码

不要做：

- 把 provider 专属行为塞进 `sync/`
- 没有明确需求就增加兼容胶水
- 把副作用藏进格式化辅助函数
- 在新的运行时策略里依赖原始错误消息文本
- 增加只有一个调用方需要的宽泛抽象
