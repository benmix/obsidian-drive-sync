# 同步初始化策略

> 生效日期：2026-03-09
> 本文只适用于建立第一份可信基线的阶段。初始化结束后，应切换到 [`SYNC_STRATEGY.md`](./SYNC_STRATEGY.md)。

## 1. 目的

初始化是一个单独阶段，因为此时系统还没有可信基线。这个阶段的目标是：

- 建立可靠的首次同步基线
- 在本地和远端状态不一致时避免误删或误覆盖
- 在执行高风险操作前把决策透明地展示出来

## 2. 进入与退出条件

### 2.1 进入初始化

满足以下任一条件时进入初始化：

1. 不存在可信的同步基线
2. 没有可靠的 `lastSyncAt`，例如首次安装或状态已被重置
3. 用户显式触发重新初始化流程

### 2.2 退出初始化

只有在以下条件全部满足时才退出初始化：

1. 初始化任务执行完成，且没有阻塞性失败
2. 已写入后续增量同步所需的可信基线
3. 已设置 `initializationCompleted` 或等价状态标记

## 3. 全局规则

优先级从高到低如下：

1. 初始化必须先做 preflight，并展示上传、下载和删除数量
2. 当本地为空且远端非空时，无论策略如何都优先执行远端恢复
3. 如果存在两个都说得通的选择，优先风险更低的一侧
4. 若没有命中硬规则，则回退到 `syncStrategy`

## 4. 决策矩阵

| 场景               | `local_win`                                            | `remote_win`                                            | `bidirectional`                                                                                   |
| ------------------ | ------------------------------------------------------ | ------------------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| 本地为空，远端非空 | 强制 `download` 与 `create-local-folder`               | 强制 `download` 与 `create-local-folder`                | 强制 `download` 与 `create-local-folder`                                                          |
| 本地非空，远端为空 | `upload` 与 `create-remote-folder`                     | 经过第二次高风险确认后执行 `delete-local`               | `upload` 与 `create-remote-folder`                                                                |
| 本地为空，远端为空 | 写空基线，不做其他事                                   | 写空基线，不做其他事                                    | 写空基线，不做其他事                                                                              |
| 双边都非空         | 对相同路径且未变化的文件仅写基线，其余遵循 `local_win` | 对相同路径且未变化的文件仅写基线，其余遵循 `remote_win` | 对相同路径且未变化的文件仅写基线，其余先用 `mtime` 做初始化阶段的时间裁决，仍无法判定时再进入冲突 |

## 5. 同路径基线捷径

这条规则只在初始化阶段生效，而且只适用于本地和远端在同一路径上都存在的文件。

判定顺序：

1. 若双方都提供 `mtime` 和 `size`，先比较规范化后的时间与大小
2. 若元数据不足以证明相同，则在初始化阶段允许做一次内容比较
3. 若内容相同，则视为未变化
4. 对未变化文件，只写基线，不生成上传、下载或冲突副本任务

这条捷径对三种策略都生效。

## 6. `bidirectional` 的时间裁决

如果上面的同路径捷径仍然无法证明相同，则 `bidirectional` 在初始化阶段额外允许一条仅限初始化的时间裁决规则。

判定规则：

1. 如果双方都提供有效 `mtime`，则比较时间
2. 如果本地较新，规划 `upload`
3. 如果远端较新，规划 `download`
4. 如果元数据缺失或仍然不足以判断，则不要自动选边，回退到冲突处理

约束：

- 这条规则不适用于文件夹
- 初始化结束后不再适用
- 它存在的目的只是减少首次同步时不必要的冲突

## 7. 本地为空时的远端恢复硬规则

触发条件：

1. 当前处于初始化阶段
2. 本地 vault 为空
3. 远端根目录不为空

必须满足的行为：

- 只允许 `download` 和 `create-local-folder`
- 明确禁止 `upload` 和 `delete-remote`
- 恢复完成后立刻写入基线
- 初始化结束后切换到运行时策略

即使初始化之后本地再次被清空，也不得重新触发这条硬规则。

## 8. Preflight 与确认

初始化执行前，preflight 必须展示：

- 上传数量
- 下载数量
- 本地删除数量
- 远端删除数量

补充规则：

- 只要存在任一删除数量，就必须进行第二次确认
- `remote_win` 下“本地非空、远端为空”属于高风险场景，默认应阻止继续，除非用户显式确认

## 9. 失败恢复

- 初始化被中断后必须可重入
- 已写入的基线与任务状态在安全前提下应被复用
- 初始化结束前不得提前进入常规增量同步
- 连续失败达到阈值后，应进入阻塞状态并要求用户介入

## 10. 与运行时同步的边界

- 本文只负责首次基线建立与显式重新初始化
- [`SYNC_STRATEGY.md`](./SYNC_STRATEGY.md) 负责基线建立后的常规同步
- 一次决策周期只能属于其中一个阶段

## 11. 验证要求

### 11.1 最低单元覆盖

- 初始化阶段检测与退出逻辑
- 本地为空时只允许下载和创建本地文件夹
- 相同路径且未变化的文件在所有策略下都只写基线
- `bidirectional` 在本地 `mtime` 更大时选择上传
- `bidirectional` 在远端 `mtime` 更大时选择下载
- 元数据不足时回退到策略特定处理
- 初始化后的本地清空不会重新触发硬规则
- `remote_win` 高风险场景需要第二次确认

### 11.2 集成覆盖

- 首次安装且远端已有数据时，能够正确恢复到本地
- 首次安装且双边都非空时，遵守策略规则
- 相同路径且规范化后元数据相同的文件只写基线
- 元数据不足但内容相同的文件只写基线
- 相同路径文件在 `bidirectional` 下会按较新的 `mtime` 选边
- 初始化被中断后可以恢复并完成

### 11.3 发布门槛

- `pnpm run lint` 通过
- `pnpm run test` 通过
- `pnpm run build` 通过
- 人工验证至少覆盖一次“本地为空、从远端恢复”的完整流程
- 人工验证至少覆盖一次“同路径文件未变化”的流程
- 人工验证至少覆盖一次“同路径文件本地与远端 `mtime` 不同”的 `bidirectional` 流程
