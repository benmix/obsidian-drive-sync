# 同步初始化策略（首次基线建立）

> 生效日期：2026-03-09
> 本文档仅约束初始化阶段的行为。
> 初始化完成后应切换到 `SYNC_STRATEGY.md`。

## 1. 目标与范围

1. 定义在没有稳定基线时的首次同步决策规则。
2. 说明初始化阶段的风险控制、预检查确认、退出条件与失败恢复行为。
3. 确保初始化不会因为策略歧义而导致大规模误删或误覆盖。

## 2. 进入与退出条件

### 2.1 进入初始化阶段

在以下任一条件成立时进入初始化：

1. 未检测到稳定同步基线（没有有效的 `entries` / `synced*` 基线）。
2. 不存在可信的 `lastSyncAt`（首次安装或状态被重置）。
3. 用户显式触发重新初始化流程。

### 2.2 退出初始化阶段

仅在以下条件全部满足时退出初始化：

1. 初始化规划出的作业已执行完成（队列为空且无阻塞失败）。
2. 同步基线已写入（未来增量决策可使用 `syncedLocalHash` / `syncedRemoteRev`）。
3. 已设置 `initializationCompleted = true`（或等价状态标记）。

## 3. 全局初始化规则（优先级从高到低）

1. 安全优先：初始化必须带 preflight，并显示上传/下载/删除计数。
2. 本地为空时优先恢复：当本地为空且远端非空时，强制执行远端到本地恢复。
3. 优先选择低风险方案：存在多个选择时，应优先采用非破坏性动作。
4. 策略回退：如果没有命中硬规则，则按 `syncStrategy` 决策。

## 4. 初始化决策矩阵

| 初始化场景         | `local_win`                                                     | `remote_win`                                                     | `bidirectional`                                                                                                       |
| ------------------ | --------------------------------------------------------------- | ---------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| 本地空，远端非空   | `download/create-local-folder`（强制）                          | `download/create-local-folder`（强制）                           | `download/create-local-folder`（强制）                                                                                |
| 本地非空，远端空   | `upload/create-remote-folder`                                   | `delete-local`（高风险，需二次确认）                             | `upload/create-remote-folder`                                                                                         |
| 本地空，远端空     | no-op（写入空基线）                                             | no-op（写入空基线）                                              | no-op（写入空基线）                                                                                                   |
| 本地非空，远端非空 | 同路径未变更文件仅写基线；否则按 `local_win` 执行并保留冲突副本 | 同路径未变更文件仅写基线；否则按 `remote_win` 执行并保留冲突副本 | 同路径文件先检查 `mtime+size`；未变更文件仅写基线；否则按 `mtime` 判断较新一侧获胜；无法判断时进入 `conflict_pending` |

## 5. 初始化阶段的同路径基线捷径

此规则仅在初始化阶段生效，并且只适用于双端都存在的同路径文件。

决策规则：

1. 如果本地和远端文件元数据都包含有效的 `mtime` 与 `size`，且 `size` 相同、`mtime` 在 provider 精度归一后视为相等，则判定该文件未变更。
2. 如果元数据无法证明相等，但双端仍存在同路径文件，则执行一次仅限初始化阶段的内容比对。
3. 如果文件内容一致，也判定为未变更。
4. 对未变更文件，只写入同步基线；不生成 `upload`、`download` 或冲突副本作业。
5. 此捷径适用于 `local_win`、`remote_win` 与 `bidirectional`。

## 6. `bidirectional` 初始化阶段的时间判定

如果上述同路径基线捷径未命中，则 `bidirectional` 会对双端同路径文件应用一个仅初始化阶段生效的时间判定规则。

决策规则：

1. 如果两侧都提供了有效的 `mtime`，则比较它们。
2. 如果本地 `mtime` 更晚，规划 `upload`，并让远端写入生成下一版 revision。
3. 如果远端 `mtime` 更晚，规划 `download`，并覆盖本地文件。
4. 如果完成上述判断所需的元数据缺失或结论不明确，则不自动选边，回退到普通 `bidirectional` 冲突流程。

约束：

1. 此规则不适用于文件夹。
2. 初始化完成后不得继续使用此规则。
3. 同路径基线捷径先用归一化元数据，必要时回退到直接内容比较；但不会把内容 hash 持久化为初始化基线的一部分。
4. `bidirectional` 的时间判定仅是初始化阶段的 tie-break；运行时 `bidirectional` 仍然保持基于冲突的语义。

## 7. 本地为空时的远端恢复（硬规则）

触发条件（必须全部满足）：

1. 当前处于初始化阶段。
2. 本地为空。
3. 远端非空。

执行要求：

1. 只允许 `download` 与 `create-local-folder`。
2. 显式禁止 `upload` 与 `delete-remote`。
3. 恢复完成后，立即写入同步基线并切换到运行时策略。

约束：

1. 此规则只适用于初始化阶段。
2. 如果初始化完成后本地再次被清空，不得重新触发此规则。

## 8. Preflight 与确认

1. 初始化执行前，preflight 必须展示：
    - 上传数量
    - 下载数量
    - 删除数量（区分本地与远端）
2. 如果任一删除数量大于 0，则必须进行二次确认。
3. 在 `remote_win` 下，“本地非空 + 远端为空”属于高风险场景。默认应阻止，只能在用户显式确认后继续。

## 9. 失败恢复与重试

1. 初始化必须可重入；中断后重新进入时，应基于已写入的 state / baseline 继续。
2. 在初始化完成前，不得切换到运行时增量路径。
3. 如果连续失败达到阈值，应进入阻塞状态并要求用户介入（认证、网络、权限等）。

## 10. 与运行时策略的边界

1. 本文档仅负责首次基线建立。
2. 运行时文档（`SYNC_STRATEGY.md`）负责基线建立之后的持续同步。
3. 两个阶段不得在同一个决策周期内同时生效；每次决策只能属于其中一个阶段。

## 11. 测试与验收标准

### 11.1 最低单元测试覆盖

1. 初始化阶段识别（进入/退出）。
2. 本地为空恢复路径只能输出 `download/create-local-folder`。
3. 所有策略下，对经归一化元数据或内容比较证明未变更的同路径文件，仅写基线，不做额外同步。
4. `bidirectional` 初始化时，本地文件 `mtime` 更晚会优先 `upload`。
5. `bidirectional` 初始化时，远端文件 `mtime` 更晚会优先 `download`。
6. 元数据不明确（例如 `mtime` 相同但 `size` 不同）时，应回退到特定策略处理。
7. 初始化完成后，本地清空不得再触发初始化硬规则。
8. `remote_win` 高风险场景必须有二次确认门槛。

### 11.2 集成验收

1. 首次安装 + 远端已有数据 -> 本地恢复正确。
2. 首次安装 + 双端都非空 -> 行为符合当前策略。
3. 首次安装 + 双端同路径文件具有等价归一化 `mtime` 与相同 `size` -> 仅写基线，不做 upload/download/冲突副本。
4. 首次安装 + 双端同路径文件元数据不明确但内容一致 -> 仅写基线，不做 upload/download/冲突副本。
5. 首次安装 + 双端同路径文件都存在 -> 在 `bidirectional` 下由较新的 `mtime` 获胜。
6. 首次安装 + 双端同路径文件元数据和内容都无法确定 -> 回退到当前策略路径。
7. 初始化中断后再次开始 -> 能继续并完成基线写入。

### 11.3 发布门槛

1. `pnpm run lint` 通过。
2. `pnpm run test` 通过。
3. `pnpm run build` 通过。
4. 手工验证至少包含一次完整的“本地为空，远端恢复到本地”的初始化流程。
5. 手工验证至少包含一次“同路径文件归一化 `mtime` 相等且 `size` 相同”的初始化流程。
6. 手工验证至少包含一次“`bidirectional` 下同路径文件 `mtime` 不同”的初始化流程。
